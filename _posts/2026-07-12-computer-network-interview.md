## 一、网络分层模型

### 1.1 OSI 七层模型

| 层次 | 名称 | 功能 | 协议/设备 |
| --- | --- | --- | --- |
| 7 | 应用层 | 为应用提供网络服务 | HTTP、FTP、DNS、SMTP |
| 6 | 表示层 | 数据格式转换、加密解密 | SSL/TLS、JPEG |
| 5 | 会话层 | 建立、管理、终止会话 | RPC、SQL |
| 4 | 传输层 | 端到端的可靠传输 | TCP、UDP |
| 3 | 网络层 | 路由选择、逻辑寻址 | IP、ICMP、路由器 |
| 2 | 数据链路层 | 帧传输、差错检测 | MAC、交换机 |
| 1 | 物理层 | 比特流传输 | 网线、光纤、集线器 |

### 1.2 TCP/IP 四层模型

| TCP/IP 层 | 对应 OSI 层 | 核心协议 |
| --- | --- | --- |
| 应用层 | 应用层+表示层+会话层 | HTTP、DNS、FTP |
| 传输层 | 传输层 | TCP、UDP |
| 网际层 | 网络层 | IP、ICMP、ARP |
| 网络接口层 | 数据链路层+物理层 | Ethernet、Wi-Fi |

### 1.3 每层都有哪些协议？（高频）

- **应用层**：HTTP/HTTPS、DNS、FTP、SMTP/POP3/IMAP、SSH、Telnet、DHCP
- **传输层**：TCP、UDP
- **网络层**：IP（IPv4/IPv6）、ICMP、ARP、RARP、OSPF、BGP
- **数据链路层**：Ethernet、PPP、VLAN

## 二、TCP 协议（重中之重）

### 2.1 TCP vs UDP

| 特性 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠传输 | 不可靠传输 |
| 有序性 | 保证有序 | 不保证有序 |
| 速度 | 较慢 | 较快 |
| 头部开销 | 20 字节 | 8 字节 |
| 传输方式 | 字节流 | 数据报 |
| 应用场景 | 文件传输、网页、邮件 | 视频、语音、DNS查询、游戏 |

### 2.2 TCP 三次握手（★★★）

```
客户端                          服务端
  |                               |
  |------- SYN (seq=x) --------->|  第一次握手：客户端发送 SYN
  |                               |
  |<--- SYN+ACK (seq=y,ack=x+1) -|  第二次握手：服务端回复 SYN+ACK
  |                               |
  |------- ACK (ack=y+1) ------->|  第三次握手：客户端发送 ACK
  |                               |
  |      连接建立，开始传输数据      |
```

**为什么是三次握手？**

- **两次不行**：如果只有两次握手，服务端无法确认客户端的接收能力。还可能出现已失效的连接请求到达服务端，导致服务端白白建立连接浪费资源。
- **四次多余**：三次已经足够双方确认彼此的发送和接收能力。

**三次握手的本质**：确认双方的发送能力和接收能力都正常。
- 第一次：服务端确认客户端的**发送能力**正常
- 第二次：客户端确认服务端的**发送能力**和**接收能力**正常
- 第三次：服务端确认客户端的**接收能力**正常

### 2.3 TCP 四次挥手（★★★）

```
客户端                          服务端
  |                               |
  |------- FIN (seq=u) --------->|  第一次挥手：客户端请求关闭
  |                               |
  |<------ ACK (ack=u+1) --------|  第二次挥手：服务端确认收到
  |                               |
  |     （服务端可能继续发送数据）    |
  |                               |
  |<------ FIN (seq=w) ----------|  第三次挥手：服务端请求关闭
  |                               |
  |------- ACK (ack=w+1) ------->|  第四次挥手：客户端确认收到
  |                               |
  |     TIME_WAIT (2MSL)          |
```

**为什么是四次挥手？**

TCP 是**全双工**通信，每个方向的连接需要单独关闭。客户端发 FIN 只是表示客户端不再发送数据了，但服务端可能还有数据要发送，所以需要分开关闭。

**为什么要 TIME_WAIT（等待 2MSL）？**

1. 确保最后一个 ACK 能到达服务端（如果丢失，服务端会重发 FIN）
2. 让本次连接的所有报文段在网络中消失，防止影响新连接

### 2.4 TCP 流量控制

**滑动窗口机制**：接收方通过 TCP 头部的**窗口大小**字段告知发送方自己还能接收多少数据。

- 发送方维护一个**发送窗口**，窗口大小由接收方决定
- 窗口内的数据可以连续发送，无需等待确认
- 当窗口为 0 时，发送方暂停发送（零窗口探测机制防止死锁）

### 2.5 TCP 拥塞控制（★★）

四个核心算法：

1. **慢启动**：cwnd 从 1 开始，每收到一个 ACK，cwnd 翻倍（指数增长），直到达到 ssthresh
2. **拥塞避免**：cwnd 超过 ssthresh 后，每个 RTT 增加 1（线性增长）
3. **快重传**：收到 3 个重复 ACK 时，立即重传丢失的报文段（不等待超时）
4. **快恢复**：快重传后，ssthresh = cwnd/2，cwnd = ssthresh，然后进入拥塞避免

```
cwnd
 ^
 |          /\
 |         /  \
 |        /    \____/\
 |       /            \___
 |      /                 \___
 |     /
 |    /
 |   /
 |  /  <- 慢启动（指数增长）
 | /
 |/________________________> time
        ^ssthresh
```

### 2.6 TCP 粘包问题

**什么是粘包？** TCP 是面向字节流的，没有消息边界的概念，多个消息可能被合并成一个 TCP 段发送（粘包）或一个消息被拆分成多个 TCP 段（拆包）。

**解决方案**：
1. **固定长度**：每个消息固定长度
2. **分隔符**：使用特殊字符（如 `\r\n`）分割消息
3. **长度前缀**：消息头部包含消息长度（最常用）
4. **自定义协议**：如 HTTP 用 Content-Length 或 chunked 编码

### 2.7 TCP 重传机制

- **超时重传**：发送方在 RTO（Retransmission Timeout）内未收到 ACK，重传数据
- **快速重传**：收到 3 个重复 ACK，立即重传
- **SACK（选择性确认）**：接收方告知发送方已收到的不连续数据段，发送方只重传缺失的段

## 三、HTTP 协议

### 3.1 HTTP 请求方法

| 方法 | 说明 | 幂等 | 安全 |
| --- | --- | --- | --- |
| GET | 获取资源 | 是 | 是 |
| POST | 提交数据 | 否 | 否 |
| PUT | 替换资源 | 是 | 否 |
| DELETE | 删除资源 | 是 | 否 |
| PATCH | 部分修改 | 否 | 否 |
| HEAD | 获取头部 | 是 | 是 |
| OPTIONS | 查询支持的方法 | 是 | 是 |

### 3.2 GET vs POST（★★★）

| 对比项 | GET | POST |
| --- | --- | --- |
| 参数位置 | URL 查询字符串 | 请求体 |
| 参数长度 | 受 URL 长度限制（浏览器限制） | 无限制 |
| 安全性 | 参数暴露在 URL 中 | 相对安全 |
| 缓存 | 可被缓存 | 一般不缓存 |
| 幂等性 | 幂等 | 非幂等 |
| 书签 | 可收藏为书签 | 不能收藏 |
| 编码 | 只支持 URL 编码 | 支持多种编码 |

**本质区别**：GET 用于获取资源，是安全且幂等的；POST 用于提交数据，可能改变服务器状态。

### 3.3 HTTP 状态码（★★）

| 分类 | 含义 | 常见状态码 |
| --- | --- | --- |
| 1xx | 信息提示 | 100 Continue |
| 2xx | 成功 | 200 OK、201 Created、204 No Content |
| 3xx | 重定向 | 301 永久重定向、302 临时重定向、304 Not Modified |
| 4xx | 客户端错误 | 400 Bad Request、401 Unauthorized、403 Forbidden、404 Not Found |
| 5xx | 服务器错误 | 500 Internal Error、502 Bad Gateway、503 Service Unavailable |

**301 vs 302**：
- 301（永久重定向）：搜索引擎会更新索引，浏览器会缓存
- 302（临时重定向）：搜索引擎不更新索引

### 3.4 HTTP 1.0 / 1.1 / 2.0 / 3.0 对比（★★）

| 版本 | 关键特性 |
| --- | --- |
| HTTP/1.0 | 每次请求都建立新的 TCP 连接（短连接） |
| HTTP/1.1 | 持久连接（默认 keep-alive）、管线化、Host 头部、分块传输 |
| HTTP/2.0 | 多路复用、头部压缩（HPACK）、服务器推送、二进制帧 |
| HTTP/3.0 | 基于 QUIC（UDP）、0-RTT 建连、改进的拥塞控制 |

**HTTP/2 多路复用 vs HTTP/1.1 管线化**：
- 管线化：请求可以并行发送，但响应必须按顺序返回（队头阻塞）
- 多路复用：请求和响应都可以乱序，通过流 ID 标识，彻底解决队头阻塞

### 3.5 HTTPS（★★★）

**HTTPS = HTTP + TLS/SSL**

**TLS 握手过程（简化版）**：

```
客户端                          服务端
  |                               |
  |------- ClientHello ---------->|  支持的 TLS 版本、加密套件、随机数
  |                               |
  |<------ ServerHello -----------|  选择的加密套件、证书、随机数
  |                               |
  |  验证证书 → 生成预主密钥       |
  |------- 用公钥加密预主密钥 ---->|  
  |                               |
  |  双方计算会话密钥（对称密钥）    |
  |                               |
  |<====== 使用对称加密通信 ======>|
```

**为什么用非对称加密 + 对称加密？**
- 非对称加密安全但慢，用来交换密钥
- 对称加密快，用来加密实际数据

**CA 证书的作用**：防止中间人攻击。CA 用自己的私钥签名服务器的公钥，客户端用 CA 的公钥验证证书真伪。

### 3.6 HTTP 缓存机制

**强缓存**（不发请求，直接用缓存）：
- `Cache-Control: max-age=3600`（HTTP/1.1，优先级高）
- `Expires: Thu, 01 Dec 2025 16:00:00 GMT`（HTTP/1.0）

**协商缓存**（发请求确认是否可用）：
- `Last-Modified` / `If-Modified-Since`：基于修改时间
- `ETag` / `If-None-Match`：基于内容哈希（优先级高）

命中协商缓存返回 **304 Not Modified**。

## 四、DNS 域名解析

### 4.1 DNS 解析过程（★★）

```
1. 浏览器缓存 → 2. 操作系统缓存（hosts 文件）
    → 3. 本地 DNS 服务器
    → 4. 根域名服务器 → 5. 顶级域名服务器 → 6. 权威域名服务器
```

- **递归查询**：客户端 → 本地 DNS（本地 DNS 负责全部查询过程）
- **迭代查询**：本地 DNS → 根/顶级/权威 DNS（每次返回下一步地址）

### 4.2 DNS 使用什么协议？

- 一般使用 **UDP**（报文小于 512 字节时）
- 区域传输使用 **TCP**（数据量大时）

## 五、IP 协议与网络层

### 5.1 IPv4 vs IPv6

| 特性 | IPv4 | IPv6 |
| --- | --- | --- |
| 地址长度 | 32 位 | 128 位 |
| 地址数量 | ~43 亿 | 几乎无限 |
| 表示方式 | 点分十进制（192.168.1.1） | 冒号十六进制 |
| NAT | 需要 | 不需要 |
| 安全性 | 可选 IPSec | 内置 IPSec |

### 5.2 IP 地址分类

| 类别 | 范围 | 默认子网掩码 | 私有地址范围 |
| --- | --- | --- | --- |
| A 类 | 1.0.0.0 ~ 126.255.255.255 | 255.0.0.0 | 10.0.0.0/8 |
| B 类 | 128.0.0.0 ~ 191.255.255.255 | 255.255.0.0 | 172.16.0.0/12 |
| C 类 | 192.0.0.0 ~ 223.255.255.255 | 255.255.255.0 | 192.168.0.0/16 |

### 5.3 ARP 协议

**ARP（Address Resolution Protocol）**：将 IP 地址解析为 MAC 地址。

工作流程：
1. 主机检查 ARP 缓存表
2. 缓存未命中 → 广播 ARP 请求（"谁是 192.168.1.1？"）
3. 目标主机单播回复自己的 MAC 地址
4. 发送方更新 ARP 缓存

### 5.4 ICMP 协议

**ICMP（Internet Control Message Protocol）**：网络诊断与差错报告协议。

常见应用：
- `ping`：测试网络连通性（ICMP Echo Request/Reply）
- `traceroute`：追踪路由路径（利用 TTL 递增）

### 5.5 子网划分与 CIDR

**CIDR（Classless Inter-Domain Routing）**：无类别域间路由。

例：`192.168.1.0/24` 表示前 24 位是网络地址，后 8 位是主机地址。

- 子网掩码：`255.255.255.0`
- 可用主机数：2^8 - 2 = 254（去掉网络地址和广播地址）

## 六、浏览器输入 URL 到页面展示（★★★）

这是面试超高频问题，涉及多个知识点的综合：

1. **URL 解析**：浏览器解析 URL，提取协议、域名、端口、路径
2. **DNS 解析**：域名 → IP 地址（浏览器缓存 → OS 缓存 → 本地 DNS → 递归查询）
3. **建立 TCP 连接**：三次握手
4. **TLS 握手**（如果是 HTTPS）：证书验证、密钥协商
5. **发送 HTTP 请求**：构建请求报文（请求行 + 请求头 + 请求体）
6. **服务器处理请求**：路由、业务逻辑、数据库查询等
7. **返回 HTTP 响应**：状态码 + 响应头 + 响应体
8. **浏览器解析渲染**：
   - 解析 HTML → DOM 树
   - 解析 CSS → CSSOM 树
   - DOM + CSSOM → Render 树
   - 布局（Layout）→ 计算元素位置和大小
   - 绘制（Paint）→ 渲染到屏幕
9. **断开连接**：四次挥手

## 七、Cookie、Session、Token

### 7.1 Cookie vs Session

| 对比项 | Cookie | Session |
| --- | --- | --- |
| 存储位置 | 客户端（浏览器） | 服务端 |
| 安全性 | 较低（可被篡改） | 较高 |
| 大小限制 | 4KB | 无明确限制 |
| 生命周期 | 可设置过期时间 | 默认浏览器关闭失效 |
| 性能 | 不占服务器资源 | 占用服务器内存 |

### 7.2 Session 的实现

1. 客户端首次请求，服务端创建 Session，生成 Session ID
2. 通过 `Set-Cookie: JSESSIONID=xxx` 返回给客户端
3. 后续请求携带 Cookie，服务端通过 Session ID 查找对应 Session

### 7.3 Token（JWT）

**JWT = Header + Payload + Signature**

优点：
- 无状态，服务器不需要存储
- 适合分布式系统和跨域认证
- 可携带用户信息

## 八、网络安全

### 8.1 常见网络攻击

**XSS（跨站脚本攻击）**：
- 存储型：恶意脚本存入数据库
- 反射型：恶意脚本在 URL 中
- 防御：输入过滤、输出转义、CSP

**CSRF（跨站请求伪造）**：
- 利用用户已登录的身份发送恶意请求
- 防御：CSRF Token、SameSite Cookie、Referer 检查

**SQL 注入**：
- 在输入中注入 SQL 语句
- 防御：参数化查询、ORM、输入验证

**中间人攻击（MITM）**：
- 攻击者位于通信双方之间，窃取或篡改数据
- 防御：HTTPS、证书固定（Certificate Pinning）

### 8.2 对称加密 vs 非对称加密

| 特性 | 对称加密 | 非对称加密 |
| --- | --- | --- |
| 密钥 | 加解密使用相同密钥 | 公钥加密，私钥解密 |
| 速度 | 快 | 慢 |
| 安全性 | 密钥分发困难 | 公钥可公开 |
| 代表算法 | AES、DES | RSA、ECC |

## 九、其他高频考点

### 9.1 WebSocket

- **全双工通信**：客户端和服务端可以同时发送数据
- **持久连接**：只需一次握手，之后保持连接
- **轻量级**：数据帧头部只有 2-10 字节
- 适用场景：实时聊天、在线游戏、股票行情

### 9.2 CDN（内容分发网络）

CDN 通过将内容部署到各地的边缘节点，让用户就近获取内容，提高访问速度。

工作原理：
1. 用户请求域名 → DNS 解析到 CDN 的智能 DNS
2. 智能 DNS 返回距离用户最近的 CDN 节点 IP
3. 用户从最近的节点获取内容

### 9.3 正向代理 vs 反向代理

| 类型 | 代理对象 | 用途 |
| --- | --- | --- |
| 正向代理 | 客户端 | 翻墙、匿名访问、缓存 |
| 反向代理 | 服务端 | 负载均衡、安全防护、SSL 卸载 |

### 9.4 负载均衡策略

- **轮询（Round Robin）**：请求依次分配给后端服务器
- **加权轮询**：根据服务器性能分配权重
- **IP 哈希**：根据客户端 IP 分配，保持会话一致性
- **最少连接**：分配给当前连接数最少的服务器

## 十、网络编程基础

### 10.1 Socket 编程

Socket 是应用层和传输层之间的抽象层，提供了进程间网络通信的接口。

**TCP Socket 编程流程**：

```
服务端                          客户端
socket()                       socket()
bind()                            |
listen()                          |
accept() ← 阻塞等待              |
  |                          connect()
  |<---- 三次握手建立连接 ---->   |
recv()/send()               send()/recv()
close()                       close()
```

### 10.2 IO 多路复用（★★）

| 方式 | 特点 |
| --- | --- |
| select | 最大 1024 个 fd，每次需要遍历所有 fd |
| poll | 无 fd 数量限制，但仍需遍历 |
| epoll | 基于事件驱动，只处理就绪的 fd，性能最优（Linux） |

**epoll 的两种模式**：
- **LT（水平触发）**：只要有数据就一直通知（默认）
- **ET（边缘触发）**：只在状态变化时通知一次（效率更高，但编程复杂）
